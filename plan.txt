Below is how I would design it as a principal engineer.

## 1. Product thesis

Build this as a **local-first fraud-defense browser agent**, not as a general browsing surveillance tool.

The extension’s job is to answer one question continuously:

> “Is the user about to trust, pay, authenticate, install, disclose, or authorize something in a context that looks unsafe?”

That means the agent should not analyze every pixel and every sentence equally. It should focus on **risk-bearing moments**:

* entering credentials, OTPs, card numbers, tax IDs, wallet seed phrases, or bank details
* clicking payment, checkout, crypto, wire, gift card, remote-support, or wallet-connect flows
* granting OAuth permissions
* downloading executables or browser updates
* interacting with popups, overlays, fake support modals, fake CAPTCHA, or fake browser/system warnings
* opening webmail messages with suspicious links, attachments, sender spoofing, or urgent payment/login language

This matters because Chrome Web Store policy is sensitive about browsing activity collection: collection and use of web browsing activity is prohibited except where required for a prominently disclosed user-facing feature. Your whole product must be explainable as “fraud protection,” with clear UI and privacy controls. ([Chrome for Developers][1])

---

## 2. Hard platform constraints

Design for **Manifest V3** from day one. MV3 moves background logic to service workers that run only when needed, removes remotely hosted executable code, and pushes request filtering toward `declarativeNetRequest` rather than unrestricted blocking request interception. ([Chrome for Developers][2])

This directly shapes the architecture:

| Need                                         | Design choice                                                                       |
| -------------------------------------------- | ----------------------------------------------------------------------------------- |
| Monitor pages                                | Content scripts with explicit host permissions or opt-in all-sites permission       |
| Block known-bad network requests             | `declarativeNetRequest` static/dynamic rules                                        |
| Runtime page analysis                        | Content script extracts redacted features, service worker orchestrates              |
| DOM-heavy work unavailable in service worker | Use content script or offscreen document where appropriate                          |
| Advanced analyzers                           | Bundle executable analyzer code in extension releases, or run analyzers server-side |
| Fast threat updates                          | Use signed JSON/rule/config updates, not remote JS/WASM                             |

MV3 remote-hosted code rules are especially important for your “tool designers can load in advanced analysis” idea. Chrome treats JavaScript/WASM loaded from outside the extension package as remotely hosted code, and MV3 requires extension code to be bundled in the extension itself. Data files like JSON are different. So the safe pattern is: **remote config and signatures are okay; remote executable analyzers are not, unless they run on your backend.** ([Chrome for Developers][3])

## 3. System architecture

### Recommended architecture

```text
Browser Extension
  ├─ Content scripts
  │   ├─ DOM observer
  │   ├─ form/action detector
  │   ├─ popup/overlay detector
  │   ├─ webmail/email extractor
  │   └─ redaction layer
  │
  ├─ MV3 service worker
  │   ├─ event router
  │   ├─ local risk scorer
  │   ├─ cache
  │   ├─ policy engine
  │   ├─ DNR rule updater
  │   └─ cloud escalation client
  │
  ├─ UI surfaces
  │   ├─ toolbar state
  │   ├─ inline warning banners
  │   ├─ interstitial block page
  │   ├─ “why this is risky” explanation
  │   └─ report false positive / false negative
  │
  └─ Local stores
      ├─ user preferences
      ├─ allowlist / blocklist
      ├─ cached domain verdicts
      └─ signed rulesets

Backend
  ├─ Analysis API
  ├─ Threat-intel aggregator
  ├─ Claude/model gateway
  ├─ Rule/config signing service
  ├─ Case review tooling
  ├─ Telemetry pipeline
  ├─ Evaluation harness
  └─ Admin / analyst console
```

The browser extension should make a fast local decision first. It should escalate to the backend only when the local score crosses a threshold or the user triggers a sensitive action.

## 4. Detection pipeline

Think of detection as a cascade, from cheap and deterministic to expensive and semantic.

### Stage 0: Pre-navigation reputation

Before or during page load:

* canonicalize URL
* check domain against local allowlist/blocklist
* check known-bad threat feeds
* check punycode/homograph risk
* detect typo-squatting against common brands
* inspect redirect chain
* inspect whether the page is newly seen by this user
* inspect risky TLD/domain patterns
* check whether the page is asking for credentials outside the expected brand domain

For commercial use, you probably want **Google Web Risk** or another commercial URL reputation provider, because the public Safe Browsing v4 docs say the API is deprecated, non-commercial, and commercial users should use Web Risk. Safe Browsing v4’s Update API is privacy-friendlier and lower-latency than Lookup because it keeps local hash-prefix data, but it is more complex to implement. ([Google for Developers][4])

### Stage 1: Page feature extraction

Content scripts should extract **features**, not raw browsing history:

```json
{
  "page": {
    "origin": "https://example.com",
    "canonical_domain": "example.com",
    "title": "...",
    "visible_text_snippets": ["..."],
    "forms": [
      {
        "intent": "login|payment|banking|crypto_seed|support|unknown",
        "fields": ["email", "password", "otp"],
        "submit_origin": "https://example.com"
      }
    ],
    "links": [
      {
        "visible_text": "Verify account",
        "href_domain": "..."
      }
    ],
    "popup_signals": {
      "modal_count": 2,
      "fullscreen_like": true,
      "system_warning_language": true
    }
  }
}
```

Avoid sending raw passwords, OTPs, card numbers, cookies, auth headers, hidden form values, or full page dumps. Redact locally before network transmission.

### Stage 2: Interaction-aware scoring

The score should increase when the user is taking a sensitive action:

| User action           | Risk trigger                                                |
| --------------------- | ----------------------------------------------------------- |
| Types password        | Is this a trusted login domain? Is this a lookalike?        |
| Types OTP             | Is the site asking for OTP after suspicious referral/email? |
| Types card/bank info  | Is checkout/payment domain trusted?                         |
| Types seed phrase     | Almost always critical risk                                 |
| Downloads executable  | Is source domain trusted?                                   |
| Clicks wallet connect | Is dApp domain known? Is request unusual?                   |
| Clicks email link     | Sender/domain/link mismatch?                                |
| Sees popup            | Fake system/browser warning? Remote-support scam language?  |

The product should be strongest at the **moment of harm**, not merely at passive browsing.

### Stage 3: LLM/Claude escalation

Use Claude for semantic interpretation when deterministic signals are ambiguous:

* “Is this email likely phishing?”
* “Does this page impersonate a bank, exchange, shipping company, employer, or government agency?”
* “Is this popup using fake support/scareware tactics?”
* “Is the user being instructed to buy gift cards, crypto, wire money, install remote desktop software, or bypass warnings?”
* “Does the message contain business email compromise indicators?”
* “Is the page asking for a seed phrase/private key?”

Use Claude through your **backend model gateway**, not directly from the extension. This protects API keys, centralizes rate limits, enables retries, enforces redaction, and lets you swap models. Anthropic’s Claude API supports tool use, where Claude can request structured tool calls that your application executes, and Anthropic documents rate and spend limits that you will need to manage centrally. ([Claude Platform Docs][5])

Prompt caching is also relevant because your system prompts, rubric, policy, and schema will be repetitive across requests; Anthropic documents prompt caching for reducing latency/cost on repeated prompt prefixes. ([Claude Platform Docs][6])

## 5. Claude/model gateway design

Do **not** send Claude arbitrary browser context as a free-form prompt. Wrap every request in a strict evidence contract.

### Input contract

```ts
type RiskEvidenceBundle = {
  request_id: string;
  event_type:
    | "page_visit"
    | "form_submit_attempt"
    | "popup_interaction"
    | "email_opened"
    | "link_clicked"
    | "download_started"
    | "wallet_interaction";

  user_action_risk: "none" | "low" | "medium" | "high" | "critical";

  page?: {
    origin: string;
    display_domain: string;
    title?: string;
    visible_text_redacted: string[];
    form_summary?: FormSummary[];
    popup_summary?: PopupSummary;
    link_summary?: LinkSummary[];
  };

  email?: {
    sender_display: string;
    sender_domain: string;
    reply_to_domain?: string;
    subject_redacted: string;
    body_text_redacted: string;
    links: Array<{
      anchor_text: string;
      href_domain: string;
      href_url_redacted: string;
    }>;
    auth_results?: {
      spf?: "pass" | "fail" | "unknown";
      dkim?: "pass" | "fail" | "unknown";
      dmarc?: "pass" | "fail" | "unknown";
    };
  };

  deterministic_signals: Array<{
    signal: string;
    severity: "info" | "low" | "medium" | "high" | "critical";
    explanation: string;
  }>;

  requested_output_schema_version: "risk_verdict_v1";
};
```

### Output contract

```ts
type RiskVerdict = {
  verdict: "safe" | "suspicious" | "dangerous" | "unknown";
  confidence: number;
  risk_categories: Array<
    | "phishing"
    | "credential_theft"
    | "payment_fraud"
    | "tech_support_scam"
    | "business_email_compromise"
    | "crypto_wallet_drain"
    | "malware_download"
    | "brand_impersonation"
    | "malicious_popup"
    | "social_engineering"
  >;
  user_message: string;
  technical_reasoning_summary: string;
  recommended_action:
    | "allow"
    | "warn"
    | "step_up_confirm"
    | "block"
    | "quarantine_email"
    | "report";
  evidence: string[];
};
```

The model should only classify and explain. It should not browse the web, follow links, execute page instructions, or generate scam content. Treat webpage/email content as hostile prompt-injection input.

## 6. Local analyzers you should build

You want multiple independent detectors, not one giant AI call.

### URL/domain analyzer

Signals:

* known-bad URL/domain
* domain age/reputation if available
* lookalike domain
* IDN/punycode tricks
* suspicious redirect chain
* misleading subdomain: `paypal.com.security-check.example.net`
* login page on unknown origin
* HTTPS absence or certificate anomalies where visible
* mismatch between email link anchor and target domain

### Brand impersonation analyzer

Inputs:

* page title
* logo alt text
* visible brand names
* domain similarity
* favicon hash if available
* login form presence

Verdict examples:

* “Looks like Microsoft login, but domain is not Microsoft-controlled.”
* “Looks like USPS delivery page, but domain is unrelated.”
* “Looks like Coinbase support, but requests seed phrase.”

### Form intent analyzer

Classify forms as:

* login
* payment
* bank transfer
* tax/identity
* crypto seed/private key
* remote support
* OAuth consent
* generic contact

The seed phrase/private key path should be near-automatic critical risk.

### Popup/scareware analyzer

Detect:

* fake browser update prompts
* fake antivirus alerts
* fake “your computer is infected”
* fake system dialog styling
* full-screen lock-in behavior
* phone-number support scams
* instructions to install AnyDesk/TeamViewer/remote desktop
* impossible claims like “FBI locked your browser”

### Email phishing analyzer

For webmail, start with Gmail/Outlook/Yahoo content scripts. Longer term, add OAuth mailbox connectors for users who want deeper protection.

Signals:

* sender display name vs sender domain mismatch
* reply-to mismatch
* SPF/DKIM/DMARC if available
* urgent credential/payment language
* suspicious links
* invoice/payment redirection
* attachment risk
* impersonated executives/vendors
* QR-code phishing if you add image analysis later

### Transaction-risk analyzer

Detect language and workflows involving:

* gift cards
* crypto deposits
* wire transfers
* Zelle/Cash App/Venmo-like irreversible payments
* “do not tell anyone”
* “act immediately”
* “refund overpayment”
* fake job/task scams
* romance/investment grooming patterns

This is where LLM analysis is particularly useful because scams often rely on narrative rather than a single bad URL.

## 7. Risk scoring model

Use a weighted scoring system, not a binary classifier.

```text
risk_score = 
  url_reputation_score
+ domain_similarity_score
+ content_social_engineering_score
+ form_sensitivity_score
+ interaction_sensitivity_score
+ email_auth_score
+ popup_behavior_score
+ user_history_anomaly_score
- trusted_domain_score
- user_allowlist_score
```

Suggested thresholds:

|  Score | UX                                    |
| -----: | ------------------------------------- |
|   0–29 | No warning                            |
|  30–49 | Passive toolbar caution               |
|  50–69 | Inline warning                        |
|  70–84 | Step-up confirmation before action    |
| 85–100 | Block/interstitial with override path |

Never make the product impossible to override. False positives will happen. The UX should make the safe action easy, but not trap the user.

## 8. Extension permissions strategy

Start narrow, then expand by explicit consent.

### MVP permissions

* `storage`
* `declarativeNetRequest`
* `scripting`
* `activeTab`
* host permissions for webmail domains you support
* optional host permissions for “protect all sites”

Chrome supports optional permissions and optional host permissions so users can grant access at runtime rather than at install time. That is a better trust posture for this product. ([Chrome for Developers][7])

Use `declarativeNetRequest` for block/allow/upgrade rules because Chrome designed it to modify requests without extensions intercepting and viewing all request content, which is better for privacy. ([Chrome for Developers][8])

Chrome also supports dynamic/session DNR rules, which is useful for updating known-bad domains and session-specific blocks without a full extension release. Dynamic rules persist across sessions and extension upgrades; session rules are cleared on browser shutdown or extension update. ([Chrome for Developers][8])

## 9. Update and extensibility model

You need two extensibility lanes.

### Lane A: Server-side analyzers

Best for:

* new scam patterns
* LLM prompt/rubric updates
* threat intelligence
* brand/domain reputation
* email classification improvements
* model routing

These can update quickly without Chrome Web Store review because they run on your infrastructure.

### Lane B: Client-side analyzers

Best for:

* DOM extraction
* form classification
* popup detection
* local redaction
* latency-critical blocking
* privacy-sensitive checks

These must be bundled into extension releases if they contain executable JS/WASM. For fast updates, use signed JSON configs, signed regex/rule packs, signed brand/domain lists, and DNR dynamic rules, but do not fetch and execute new code remotely. ([Chrome for Developers][3])

A good design is:

```text
Analyzer SDK
  ├─ Extractors: produce normalized evidence
  ├─ Detectors: score specific risks
  ├─ Policies: map scores to UX
  ├─ Redactors: remove secrets/PII
  └─ Explainability: produce user-facing reasons
```

Every analyzer should emit:

```ts
{
  id: "seed_phrase_detector",
  version: "1.3.0",
  score: 95,
  severity: "critical",
  evidence: ["Page asks for 12/24 recovery words"],
  redacted_fields_touched: ["textarea"],
  recommended_action: "block"
}
```

## 10. Privacy and trust design

This product will live or die on trust.

### Principles

* Local-first by default.
* Never collect full browsing history.
* Never collect passwords, OTPs, cookies, auth headers, hidden fields, private keys, or seed phrases.
* Redact before cloud.
* Let users see “what was analyzed.”
* Let users disable analysis on a site.
* Separate security telemetry from product analytics.
* Retain risky evidence only with explicit user report/consent.
* Encrypt all data in transit and at rest.
* Sign all rule/config updates.
* Keep a public privacy policy and in-product disclosure.

Chrome’s messaging security guidance is also relevant: content scripts are less trustworthy than the extension service worker, so messages from content scripts must be validated and sanitized, and privileged actions should be limited. ([Chrome for Developers][9])

## 11. User experience

The UX should be calm, specific, and actionable.

Bad warning:

> “Dangerous website detected.”

Better warning:

> “This page looks like a Microsoft login, but it is not on a Microsoft-owned domain. Do not enter your password here.”

For high-risk cases:

```text
Stop — this looks like a scam.

Why:
1. This page is asking for your wallet recovery phrase.
2. Legitimate wallet providers never ask for recovery phrases on websites.
3. The domain is not associated with the wallet brand.

Recommended action:
Close this page.

[Close page] [Report site] [I understand, continue]
```

For email:

```text
Suspicious email

This email claims to be from your payroll provider, but the link goes to an unrelated domain. The message also uses urgent account-lock language.

Recommended action:
Do not click the link. Visit the payroll site directly from your bookmark or search.
```

Avoid fear-mongering. Scammers already use fear. Your product should create clarity. Inform with explainable risk ratings (show how it was added / calculated)

## 15. Biggest risks

### Risk 1: Becoming spyware-like

Mitigation: local-first, explicit purpose, opt-in all-sites permission, visible UI, data minimization.

### Risk 2: Too many false positives

Mitigation: warnings only at sensitive action moments; use step-up risk numbers and confirmations instead of constant banners.

### Risk 3: LLM prompt injection

Mitigation: strict evidence schema, hostile-content delimiters, no autonomous browsing/tool execution, model output validation.

### Risk 4: Remote analyzer updates violate MV3

Mitigation: remote JSON/config only; executable code bundled in extension releases; server-side analyzers for rapid iteration.

### Risk 5: Latency

Mitigation: local deterministic checks first, cache verdicts, use Claude only on ambiguous/high-risk events, use prompt caching where applicable.

### Risk 6: Incomplete threat feeds

Mitigation: use multiple signals. URL reputation is necessary but insufficient; many scams are new, compromised, or socially engineered.

---

## 16. My recommended north-star architecture

The best version of this product is:

> A privacy-preserving browser safety layer that watches for high-risk user intent, extracts minimal redacted evidence, runs local deterministic detectors first, hits various free risk apis, escalates ambiguous cases to a controlled AI risk service (or a temp like sonet model for now), and gives users clear, specific, reversible warnings and risk ratings before harm occurs.

The core principle: **the user should feel protected, and informed with a risk score about each site the visit.**
